Feature March 9, 2026

Automatic ICAO / FAA Identifier Resolution

All identifier-based endpoints now automatically resolve between ICAO and FAA airport identifier formats, so you get data back regardless of which format you use.

Background

US airports have two identifier systems. FAA identifiers are the domestic format — typically 3 characters (e.g., DFW, BNA) but sometimes alphanumeric (e.g., W05, 9D4, A88). ICAO codes are the international standard — always 4 characters with a regional prefix:

Region ICAO Prefix Example
Contiguous US (lower 48) K KDFW = FAA DFW
Alaska PA PAFA = FAA AFA
Hawaii PH PHNL = FAA HNL
Guam PG PGUM = FAA GUM

Many smaller US airports only have an FAA identifier — they have no officially assigned ICAO code. Previously, if you requested data using the ICAO format for one of these airports (e.g., /airports/KW05), the API returned a 404 even though the airport exists under its FAA identifier W05.

What Changed

When you request data by airport identifier, the API now tries the alternate format automatically if the exact identifier isn't found:

/* Strip ICAO prefix → try FAA identifier */
KW05 → also tries W05
K9D4 → also tries 9D4
PA88 → also tries A88
PHNL → also tries HNL

/* Add ICAO prefix → try ICAO code */
DFW  → also tries KDFW
W05  → also tries KW05
9D4  → also tries K9D4
Before
GET /airports/KW05  404
GET /runways/airport/K9D4  404
GET /airports/K9G8  404
After
GET /airports/KW05  200
GET /runways/airport/K9D4  200
GET /airports/K9G8  200

Affected Endpoints

Identifier resolution applies to every endpoint that accepts an airport code or identifier:

  • GET /api/v1/airports/{icaoCodeOrIdent}
  • GET /api/v1/airports/batch?ids=...
  • GET /api/v1/metars/{icaoCodeOrIdent}
  • GET /api/v1/metars/batch?ids=...
  • GET /api/v1/tafs/{icaoCodeOrIdent}
  • GET /api/v1/tafs/batch?ids=...
  • GET /api/v1/runways/airport/{icaoCodeOrIdent}
  • GET /api/v1/terminal-procedures/{icaoCodeOrIdent}
  • GET /api/v1/chart-supplements/{icaoCodeOrIdent}
  • GET /api/v1/communication-frequencies/{servicedFacility}
  • GET /api/v1/e6b/crosswind/{icaoCodeOrIdent}
  • GET /api/v1/e6b/density-altitude/{icaoCodeOrIdent}
  • GET /api/v1/pireps/airport/{icaoCodeOrIdent}
  • GET /api/v1/obstacles/airport/{icaoCodeOrIdent}
  • POST /api/v1/briefing/route airport waypoints

What This Means For You

No breaking changes

Existing requests continue to work exactly as before. This is purely additive — the API now finds airports it previously missed.

  • No client-side normalization needed. If you've been manually stripping or adding prefixes before calling the API, you no longer need to. Send whichever format you have and the API will resolve it.
  • Batch requests work too. You can mix ICAO codes and FAA identifiers freely in batch endpoints (e.g., ?ids=KDFW,W05,PA88).
  • Case-insensitive. All identifier lookups are case-insensitive — kdfw, KDFW, and Kdfw all resolve the same.
Weather endpoints for small airports

Identifier resolution means the airport will be found correctly, but many small airports don't have weather reporting stations. METAR and TAF endpoints will still return 404 for these airports — the identifier resolves, but there's simply no weather data to return.

Example

Requesting airport data for Potomac Airfield (FAA identifier: VKX, no assigned ICAO code):

# All three resolve to the same airport:
GET /api/v1/airports/VKX    # FAA identifier
GET /api/v1/airports/KVKX   # ICAO format (auto-resolved)
GET /api/v1/airports/kvkx   # case-insensitive

# Related data also resolves:
GET /api/v1/runways/airport/KVKX
GET /api/v1/chart-supplements/KVKX
GET /api/v1/terminal-procedures/KVKX
GET /api/v1/communication-frequencies/KVKX
